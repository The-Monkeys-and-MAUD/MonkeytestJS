Philosophy

You might be writing unit tests, but unless a project is large enough to warrant a dedicated test team, there probably won't be any automated top level testing of the interface. This is sad because you get a lot of value from even a few high level functional tests.

Why, when there are plenty of tools around for writing automated browser based tests (like Selenium, Sahi, eValid, Watir and Canoo), do so few front end developers use them? We thought about it and decided that the existing tools just didn't suit how we worked. They're not hard to use exactly, but something about them just seems to take us too far out of the zone. Also, they're slow and there's never any budget for automated testing.

So we made MonkeyTestJS. It's quick and easy to use, so we can build up a suite of regression tests while we're building the functionality of our project. We write tests using straightforward client side javascript and run them in a browser to see the result.

MonkeyTestJS is built on top of QUnit, the unit testing framework used by JQuery. However, instead of focusing on testing javascript functions, MonkeyTestJS tests act on the actual DOM of your web page. Using standard QUnit assertions, your tests simulate user interaction and then check the DOM for expected results.

Don't take our word for it, have a look:

http://monkeytestjs.io/tests

Because the test suite is run from a URL, it can be run in a browser tab while you're working. Or in a headless browser like PhantomJS as part of continuous integration (see grunt-MonkeytestJS). Or across a suite of browsers using a service like BrowserStack.

Tests can be asynchronous so it's good for AJAX and dynamically generated markup. MonkeyTestJS also comes with a PHP proxy script for those situations where you need cross domain requests.

Custom tests are quick to write, but it's also easy to re-use tests across pages. You can test things like HTML validation, Google Analytics, character encoding etc. on all pages of your site without writing a test.
